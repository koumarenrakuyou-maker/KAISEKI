# KAISEKI — 次世代日本語言語解析システム 設計書

> LIWCの辞書ベース手法とLLMの文脈理解を融合し、日本語特有の言語文化を深く捉える新しい心理言語解析システム

---

# 1. 開発フェーズ

## Phase 0: 基盤調査・要件定義（1〜2ヶ月）
| # | タスク | 詳細 |
|:---:|:---|:---|
| 0-1 | 既存システムの技術仕様の精査 | LIWC-22, J-LIWC2015, Empath, Coh-Metrix等の内部辞書構造・アルゴリズムの完全な理解 |
| 0-2 | 日本語言語学リソースの棚卸し | 形態素解析器（MeCab, Sudachi, GiNZA, Janome）、感情辞書、日本語ワードネット、オノマトペ辞書等 |
| 0-3 | 心理学カテゴリの学術レビュー | LIWC-22の117カテゴリの妥当性検討＋日本語文化固有のカテゴリ候補の文献調査 |
| 0-4 | ユースケース定義 | 臨床心理、社会心理、認知科学、マーケティング、教育等の利用場面 |
| 0-5 | LLMモデルの選定調査 | GPT-4, Claude, Gemini または日本語特化モデル（ELYZA, rinna等）の精度・コスト比較 |

# 3. 現状考えられる質問・検討事項（20項目）

## 技術的な質問

| # | 質問 | 背景 |
|:---:|:---|:---|
| 1 | **LLMのコスト**: LLMのAPI呼び出しは1テキストあたりどの程度のコスト・時間がかかるか？ 大量分析（数千テキスト）で実用的なコストに収まるか？ | LIWC-22はローカル辞書照合のため高速・無料。AI利用は速度・コストのトレードオフが生じる |
| 2 | **LLMの再現性**: LLMは確率的な出力をするが、同じテキストに対して毎回同じ分析結果を保証できるか？ 研究利用に必要な再現性をどう担保するか？ | 科学的研究には再現性が不可欠。temperature=0でも完全一致は保証されない |
| 3 | **形態素解析アンサンブルの判定基準**: MeCab, Sudachi, GiNZAの出力が異なる場合、どの結果を採用するか？ 多数決か、AIが判定するか？ | 形態素解析器ごとに辞書・分割方針が異なる |
| 4 | **辞書のライセンス**: J-LIWC辞書をベースに拡張する場合、ライセンス上の問題はないか？ 完全独自辞書を一から構築すべきか？ | J-LIWCはLIWCライセンスが必要。独自辞書なら自由だが構築コストが大きい |
| 5 | **オフライン動作**: LLMをクラウドAPIに依存する場合、オフライン環境で動作しないが、オンプレミスの小型LLMで代替可能か？ | 医療・教育現場ではネットワーク制約がある |

## カテゴリ設計の質問

| # | 質問 | 背景 |
|:---:|:---|:---|
| 6 | **カテゴリ数の最適解**: 150+カテゴリは多すぎないか？ 多重比較の問題にどう対処するか？ | カテゴリが増えるほど分析の幅は広がるが、統計的な偽陽性リスクが増大 |
| 7 | **感情強度の基準**: 「激怒=0.95, いらだち=0.3」等のスケーリングは何を根拠とするか？ 心理学的に妥当な基準値をどう設定するか？ | 主観的な設定では研究としての妥当性が疑問視される |
| 8 | **日本文化特有カテゴリの妥当性**: 「忖度」「世間体」などを独立カテゴリとする心理学的根拠は十分か？ 実証研究で確認が必要ではないか？ | 文化心理学で議論されるが、テキスト分析での操作的定義は未確立 |
| 9 | **カテゴリの動的生成**: 研究目的ごとにAIが新カテゴリを生成する機能は、再現性と標準化の観点で問題にならないか？ | 研究間の比較可能性が失われるリスク |
| 10 | **既存カテゴリの正当性検証**: J-LIWCのカテゴリが英語から直訳されただけで日本語文脈で本当に妥当なのか、どう確認するか？ | 翻訳等価性の問題 |

## 研究方法論の質問

| # | 質問 | 背景 |
|:---:|:---|:---|
| 11 | **最低語数の緩和**: 4000語から減らす場合、信頼性低下をどう定量化するか？ 100語でも使えるようにすべきか？ | SNS投稿のような短文分析のニーズが高い |
| 12 | **妥当性検証実験のデザイン**: 感情正解文テストで使う正解ラベルは誰がつけるか？ 合意形成の方法は？ | 人間の感情判断自体にばらつきがあり、ゴールドスタンダードの設定が困難 |
| 13 | **生理指標との相関**: どの生理指標（心拍変動、皮膚電位、fMRI等）を優先して検証するか？ | すべてを同時に検証するのはコストが高い |
| 14 | **倫理審査**: 音声入力＋AI分析は個人情報の取り扱いとして、IRBでどう審査されるか？ | 音声データの保存と分析には特別な倫理的配慮が必要 |
| 15 | **他の日本語テキストマイニングツールとの比較**: KH Coder, TTM, TermExtract等と何を基準に比較するか？ | 先行ツールとの差別化と優位性の実証が必要 |

## 実装・運用の質問

| # | 質問 | 背景 |
|:---:|:---|:---|
| 16 | **ターゲットユーザーの技術レベル**: 主な利用者は心理学者・臨床家か、計算言語学者か？ UIの複雑さをどこに設定するか？ | LIWC-22のGUI並みの使いやすさか、Rコード主体か |
| 17 | **音声入力の言語変種**: 方言、なまり、吃音等の音声はどこまで対応するか？ | 臨床場面では標準語でない音声が多い |
| 18 | **バージョニング戦略**: 辞書やカテゴリを更新した際に、過去の研究結果との互換性をどう維持するか？ | LIWCも2001→2007→2015→22で辞書が変わり過去研究との比較が課題 |
| 19 | **マルチプラットフォーム**: Windows, Mac, Linux, Webすべてに対応するか？ 開発リソースの制約は？ | Rパッケージならクロスプラットフォームだが、GUIはプラットフォーム依存 |
| 20 | **OSSか商用か**: オープンソースとして公開し学術コミュニティに貢献するか、有償ライセンスとするか？ | LIWCは有償。J-LIWCは学術利用無料。公開方針が普及に直結 |

# 4. KAISEKIの利点（20項目）

| # | 利点 | 詳細 |
|:---:|:---|:---|
| 1 | **AI文脈理解による質的飛躍** | LLMにより「嬉しくない」「皮肉を言う」等の文脈依存表現を正確に解析。辞書ベースの限界を根本的に解消 |
| 2 | **日本語文化特有カテゴリの搭載** | 忖度、世間体、ウチ・ソト意識、敬語レベルなど、日本語文化固有の心理概念を初めて体系化 |
| 3 | **感情強度のスケーリング** | 「怒り」を一律カウントではなく0〜1の連続値で出力。「激怒」と「いらだち」を区別可能 |
| 4 | **否定スコープの正確な処理** | AIが否定語の影響範囲を判定し、感情極性を適切に反転。「楽しくなかった」を正しく負の感情と判定 |
| 5 | **自動辞書拡張・新語対応** | SNSやニュースコーパスから新語を自動検出し、カテゴリ推奨を提示。辞書の陳腐化を防止 |
| 6 | **形態素解析のハイブリッド方式** | 複数解析器のアンサンブル＋AI補正により、単一解析器への依存リスクを軽減 |
| 7 | **音声入力対応** | 音声→テキスト変換＋韻律分析により、面接・カウンセリング場面で直接利用可能 |
| 8 | **外部データとの統合分析** | 質問紙（BDI, GHQ等）、生理指標（HRV, EDA等）、行動データとの相関分析をシステム内で完結 |
| 9 | **信頼性レベルの自動表示** | 入力語数に応じた信頼区間を自動算出。「500語→信頼度中」「4000語→信頼度高」等を明示 |
| 10 | **初心者フレンドリーな設計** | 分析目的を選ぶだけで適切なカテゴリ・設定をAIが自動推奨。分かち書き・カテゴリ設定を自動化 |
| 11 | **オノマトペの体系的分析** | 擬音語・擬態語を感情・知覚カテゴリに自動分類。日本語表現の豊かさを活かした分析を実現 |
| 12 | **主語省略の推定機能** | 日本語で頻出する主語省略を構文解析＋AIで推定。英語の代名詞研究に相当する日本語版指標を実現 |
| 13 | **150以上のカテゴリ** | LIWC-22の117、J-LIWCの69を超える自前カテゴリ構成で、研究可能な課題の幅を大幅に拡大 |
| 14 | **多表記性の統一処理** | 漢字/ひらがな/カタカナの表記ゆれを自動統一した上で分析。「嬉しい」「うれしい」を同一語として処理 |
| 15 | **構文レベルの分析** | 単純な単語カウントではなく、係り受け・文構造を考慮した意味分析を提供 |
| 16 | **文末表現・終助詞の心理分析** | 「〜ね」「〜よ」「〜かな」等の微妙なニュアンスを心理カテゴリとして分類 |
| 17 | **AIスペルチェック・前処理自動化** | 誤字脱字、略語展開、表記統一をAIが自動処理。人手による前処理の負担を大幅に軽減 |
| 18 | **研究目的別のカテゴリ推奨** | 臨床、社会心理、認知科学など研究分野に応じた分析プロファイルの自動設定 |
| 19 | **要約変数のオープンアルゴリズム** | LIWCのブラックボックスとは異なり、すべての算出過程を公開・検証可能にする |
| 20 | **可視化ダッシュボード** | 分析結果をインタラクティブに閲覧できるWebダッシュボード。感情遷移の時系列グラフ等を標準搭載 |

| # | 限界点 | 詳細・対策の方向性 |
|:---:|:---|:---|
| 1 | **LLMのコストと速度** | LLM API呼び出しはLIWCの辞書照合に比べ大幅に遅く高コスト。大量データ分析時にボトルネックとなりうる。→ キャッシュ戦略、バッチ最適化、重要度に応じたLLM呼び出し頻度調整 |
| 2 | **再現性の課題** | LLMの確率的出力により、同一テキストへの分析結果が微小に変動しうる。→ temperature=0設定、seed固定、辞書ベース結果との併用で安定層を確保 |
| 3 | **カテゴリの妥当性が未検証** | 新規カテゴリ（忖度、世間体等）は心理学的に興味深いが、実証的検証がまだ行われていない。→ Phase4の検証実験で対処 |
| 4 | **辞書構築の初期コスト** | 15,000語以上の辞書＋150カテゴリを一から構築・検証するには大きな人的・時間的コストがかかる。→ 既存リソース活用＋AI支援による半自動構築 |
| 5 | **LLMブラックボックス問題** | LLMの判断過程は説明困難であり、「なぜこの分類をしたか」が不透明。→ Chain-of-Thought、根拠提示機能で一定の透明性を確保 |
| 6 | **インターネット接続依存** | クラウドLLMを使う場合、オフライン環境では一部機能が利用不能。→ ローカルLLM（量子化モデル）のフォールバック |
| 7 | **感情強度の基準値問題** | 0〜1スケーリングの基準設定が主観的になりうる。→ 大規模アノテーション実験で実証的に校正 |
| 8 | **方言・口語の網羅性** | 日本語の方言は多様であり、すべてを網羅するのは困難。→ 主要方言から段階的に対応 |
| 9 | **音声認識の精度依存** | 音声入力機能はWhisper等の精度に依存し、背景雑音や不明瞭な発話では精度低下。→ 音声品質チェック＋手動修正UIの提供 |
| 10 | **多重比較問題** | 150以上のカテゴリを一度に検定すると、偽陽性のリスクが増大。→ 調整済みp値の推奨、効果量の重視 |
| 11 | **プライバシーリスク** | テキスト・音声データをクラウドに送信する場合の個人情報漏洩リスク。→ PII自動マスキング、ローカル処理オプション |
| 12 | **LLMモデルの陳腐化** | 依存するLLMのサービス終了や仕様変更により、長期的な安定性が保証されない。→ モデル抽象化レイヤーで差し替え可能な設計 |
| 13 | **他システムとのベンチマーク不足** | 開発初期段階では、KH Coder, TTM等の既存ツールとの系統的な比較データが不足。→ Phase4で計画的に比較実験 |
| 14 | **古典文・文語体の非対応** | 現代日本語に最適化しているため、古文・漢文・文語体の分析精度は低い。→ 将来の拡張として検討 |
| 15 | **研究コミュニティの採用ハードル** | LIWCは30年の実績と数千の引用論文があり、新システムへの移行には大きな慣性がある。→ LIWCとの互換出力モードで移行障壁を低減 |
| 16 | **文化特有カテゴリの他文化適用不可** | 忖度・世間体カテゴリは日本語専用であり、異文化比較研究では直接使用できない。→ 文化共通カテゴリと文化固有カテゴリを明確に分離 |
| 17 | **動的辞書更新の品質管理** | 自動検出された新語の品質・カテゴリ割当ての精度保証が困難。→ 人間レビュープロセスの組み込み |
| 18 | **計算資源の要求** | 形態素解析アンサンブル＋LLM＋構文解析の組み合わせは、一般的なPCでは負荷が高い。→ 分析レベルの選択機能（軽量/標準/フル） |
| 19 | **長文テキストのセグメンテーション** | 小説や長時間インタビューなど長大なテキストの分割方針が未確定。→ 自動・手動セグメンテーション選択 |
| 20 | **日本語以外への拡張の困難** | 日本語に深く特化した設計のため、他言語への横展開は再設計が必要。→ 言語非依存コアと言語固有モジュールの分離設計 |
