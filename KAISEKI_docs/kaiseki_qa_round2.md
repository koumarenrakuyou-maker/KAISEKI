## Q1: カテゴリ数 — ゼロベースは厳しいか？

**完全ゼロベースは非現実的ですが、「LIWCリファレンス＋日本語独自設計」のハイブリッドが最善です。**

### 推奨: 段階的カテゴリ構築

| 段階 | カテゴリ数 | 内容 |
|:---|:---:|:---|
| **v0.1（MVP・卒論用）** | 30〜40 | 感情系（喜怒哀楽+α）、認知系、社会系の核心カテゴリのみ |
| **v1.0（論文投稿用）** | 80〜100 | LIWC主要カテゴリの日本語版＋日本語固有カテゴリ |
| **v2.0（本格版）** | 120〜150 | 全カテゴリ＋カスタムカテゴリ機能 |

### v0.1 (MVP) のカテゴリ候補（約35個）

```
■ 感情 (10)
  ├─ ポジティブ感情（全体）
  ├─ 喜び / 楽しさ
  ├─ 感謝
  ├─ ネガティブ感情（全体）
  ├─ 怒り
  ├─ 悲しみ
  ├─ 不安 / 恐怖
  ├─ 嫌悪
  ├─ 驚き
  └─ 感情語全体

■ 認知プロセス (6)
  ├─ 洞察（わかる、気づく等）
  ├─ 因果（なぜなら、だから等）
  ├─ 確信（絶対、必ず等）
  ├─ 不確実（たぶん、かもしれない等）
  ├─ 比較（より、ほど等）
  └─ 認知語全体

■ 社会 (5)
  ├─ 家族関連語
  ├─ 友人関連語
  ├─ 社会的関係全体
  ├─ コミュニケーション（言う、話す等）
  └─ 敬語使用

■ 人称・自己参照 (4)
  ├─ 一人称（わたし、僕、俺等）
  ├─ 二人称（あなた、君等）
  ├─ 三人称（彼、彼女等）
  └─ 代名詞全体

■ 言語構造 (6)
  ├─ 否定表現
  ├─ 疑問表現
  ├─ 終助詞（ね、よ、かな等）
  ├─ オノマトペ
  ├─ 接続詞
  └─ 句読点頻度

■ 基本統計 (4)
  ├─ 総語数
  ├─ 文の数
  ├─ 一文あたりの平均語数
  └─ 辞書カバレッジ率
```

ゼロから分類基準を決めるのではなく、**LIWCで有効性が実証されたカテゴリを出発点** としつつ、日本語の特徴に合わせて追加・修正する。これが最も効率的で学術的にも守りやすいアプローチです。

## Q2: 辞書構築方法の比較

| 方法 | 利点 | 限界 | デメリット | 推奨度 |
|:---|:---|:---|:---|:---:|
| **(A) 大規模コーパスから頻出語抽出** (BCCWJ等) | 実際の使用頻度に基づく、網羅性が高い | カテゴリ割当ては人手が必要、時間がかかる | 頻出語=心理的に重要とは限らない | ★★★ |
| **(B) 既存の感情辞書を統合** (感情極性辞書、日本語ワードネット等) | 既にカテゴリ分けされている、すぐ使える、無料 | カテゴリ体系が異なるため変換が必要、カバー範囲が感情中心 | 辞書ごとに品質がばらつく | ★★★★ |
| **(C) AIで生成して人間がレビュー** | 高速、大量生成可能、カバー範囲が広い | 品質のばらつき、幻覚（存在しない語の生成） | レビューコストが侮れない | ★★★ |
| **(D) LIWC辞書を参考に独自再構築** | 学術的に検証済みの体系をベースにできる | LIWCライセンスに触れないよう注意が必要 | 英語ベースの限界を引き継ぐリスク | ★★★ |

### **推奨: B → A → C の三段階ハイブリッド**

```
Step 1: 既存辞書の統合（即座に開始可能）
  ├─ 日本語感情極性辞書（東北大学乾研、約5,500語）→ 感情カテゴリ
  ├─ 日本語ワードネット（NICT、約57,000語）→ 意味関係
  └─ オノマトペ辞書 → オノマトペカテゴリ
     ↓
Step 2: BCCWJコーパスから頻出語を追加（辞書の隙間を埋める）
  └─ 心理関連のない高頻度語を構造カテゴリに割当て
     ↓
Step 3: AIで候補を生成し人間がレビュー（最終拡充）
  └─ 「認知語のリストを100個生成して」→ 人間が取捨選択
```

## Q3: 開発言語 — 推奨

### **推奨: Python (FastAPI) + Streamlit (Web UI)**

| 要素 | 選択 | 理由 |
|:---|:---|:---|
| **コアエンジン** | Python | NLP生態系が最も充実(spaCy, GiNZA, Transformers)。1-2人開発に最適 |
| **API層** | FastAPI | 高速・非同期・型ヒントで堅牢。LLM APIとの統合が容易 |
| **Web UI** | Streamlit | Pythonだけで素早くUIが作れる。研究者に最も人気。1人でも構築可能 |
| **R連携** | reticulate経由 | R利用者向けにPythonを呼び出すラッパー。将来的に提供 |
| **形態素解析** | GiNZA (spaCy系) | Sudachi内蔵、係り受けもできる。Python完結 |

**Streamlitを選ぶ理由:** React/Next.jsはフロントエンド開発が別スキルセットで、1-2人体制では負担が大きい。Streamlitなら**Pythonだけ**でインタラクティブなWebダッシュボードが作れる。

## Q5: 統計的検定力分析 — 必要被験者数

### 実験デザインの想定

```
デザイン: 被験者内（Within-subjects）反復測定
独立変数: 映画の感情条件（例: 楽しい / 悲しい / 怒り / 中立 = 4条件）
従属変数: KAISEKIの各感情カテゴリスコア + BioPac生理指標
```

### G*Power計算結果

| パラメータ | 設定値 |
|:---|:---|
| 検定 | 反復測定ANOVA (within factors) |
| 効果量 (Cohen's f) | 0.25（中程度） |
| α（有意水準） | 0.05 |
| 検定力 (1-β) | 0.80（80%） |
| 群数 | 1（被験者内デザイン） |
| 測定回数 | 4条件 |
| 反復測定間の相関 | 0.5（保守的推定） |
| 非球面性補正 (ε) | 0.7（保守的推定） |

### 結果

| 効果量 | 必要被験者数 | 実用的推奨 |
|:---:|:---:|:---|
| 大 (f=0.40) | **15〜20名** | 最低限。パイロット研究向け |
| **中 (f=0.25)** | **30〜36名** | **標準的な卒論水準。推奨** |
| 小 (f=0.10) | 100名以上 | 大規模研究。卒論では困難 |

### **推奨: 36名（脱落考慮で40名募集）**

- 中程度の効果量で検定力80%を確保
- 卒論として現実的な人数
- BioPac測定を含む場合、1名あたり約1〜1.5時間

> **補足**: もし2条件（楽しい vs 悲しい）に絞れば**対応のあるt検定**となり、中程度の効果量(d=0.5)で**約27名**で足りる。条件数を減らすと負担が軽くなる。
